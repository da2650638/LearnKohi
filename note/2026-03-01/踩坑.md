# 2026-03-01 踩坑总结

# 踩坑总结：DLL 导出函数实现文件没有包含声明头文件，导致符号未导出

## 1. 问题现象

编译 DLL 成功，但在链接使用该 DLL 的程序时出现错误：

```
error LNK2019: 无法解析的外部符号 __imp_report_assertion_failure
fatal error LNK1120: 1 个无法解析的外部命令
```

关键特征：

```
__imp_report_assertion_failure
```

说明调用方认为这是 **DLL 导入函数 (`dllimport`)**。

---

# 2. 问题代码结构

### asserts.h

```c
KAPI void report_assertion_failure(const char* expression,
                                   const char* message,
                                   const char* file,
                                   i32 line);
```

其中

```
KAPI -> __declspec(dllexport) / __declspec(dllimport)
```

---

### logger.c（错误写法）

```c
#include "logger.h"

void report_assertion_failure(...)   // ❌ 没有 KAPI
{
    logger_output(...);
}
```

并且 **logger.c 没有**

```
#include "asserts.h"
```

---

# 3. 根本原因

在 Windows 中：

```
__declspec(dllexport)
```

是 **编译期属性**。

编译器必须在 **看到函数定义之前** 就知道这个函数需要导出。

但是：

```
logger.c
```

没有包含 `asserts.h`，因此编译器看到的是：

```
void report_assertion_failure(...)
```

一个 **普通函数定义**。

结果：

* 该函数不会被导出到 `engine.dll`
* `engine.lib` 中也没有对应符号
* 调用方（testbed）因为 `dllimport` 期望导入
* 链接失败

---

# 4. 为什么在定义处加 `KAPI` 就能解决

```c
KAPI void report_assertion_failure(...)
```

这样做等价于：

```
__declspec(dllexport) void report_assertion_failure(...)
```

即使没有包含 `asserts.h`，编译器也知道要导出该函数。

---

# 5. 正确写法（推荐）

实现文件必须包含声明头文件：

```c
// logger.c
#include "logger.h"
#include "asserts.h"

void report_assertion_failure(...)
{
    ...
}
```

这样编译器在定义前就已经看到：

```
KAPI void report_assertion_failure(...)
```

就会正确导出。

---

# 6. 最佳实践（非常重要）

### 实现文件必须包含自己的头文件

规范写法：

```c
// asserts.c
#include "asserts.h"
```

好处：

1. 保证声明和定义一致
2. 防止忘记导出属性
3. 编译器可以检查函数签名
4. 避免 DLL 导出错误

---

# 7. 推荐的模块结构

更好的结构应该是：

```
engine/
 ├─ core/
 │   ├─ logger.h
 │   ├─ logger.c
 │   ├─ asserts.h
 │   ├─ asserts.c
```

而不是把 `asserts` 的实现写进 `logger.c`。

---

# 8. 如何快速判断是否是 DLL 导出问题

如果看到错误：

```
__imp_XXXX
```

几乎可以确定是：

```
dllimport -> dll 没导出
```

检查：

1. 是否使用 `__declspec(dllexport)`
2. 是否包含正确头文件
3. import library 是否正确链接
4. 是否使用 `.def` 或 export flag

---

# 9. 一句话总结这个坑

**DLL 函数实现文件如果没有包含带有 `dllexport` 的声明头文件，编译器不会导出该函数，从而导致调用方链接失败。**

---

# 10. 一个工程经验（强烈建议）

**每个 `.c/.cpp` 文件的第一行都包含自己的头文件**

例如：

```c
#include "asserts.h"
```

这样如果函数签名或宏不一致，**编译阶段就会报错**，而不是拖到链接阶段。
